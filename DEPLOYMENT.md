# 🚀 Deploying EduBoard

Architecture: **Frontend → Vercel**, **Backend → Render**, **Database → MongoDB Atlas**.

```
Browser ──> Vercel (React SPA) ──HTTPS /api──> Render (Express API) ──> MongoDB Atlas
```

---

## Step 1 — Database: MongoDB Atlas (free)

1. Create an account at <https://www.mongodb.com/atlas> and make a free **M0** cluster.
2. **Database Access** → add a user (username + password). Save them.
3. **Network Access** → Add IP Address → **Allow access from anywhere** (`0.0.0.0/0`).
   Render uses dynamic IPs, so this is required.
4. **Connect → Drivers** → copy the connection string. It looks like:
   ```
   mongodb+srv://<user>:<password>@cluster0.xxxxx.mongodb.net/eduboard?retryWrites=true&w=majority
   ```
   Replace `<user>`/`<password>` and keep `/eduboard` as the database name.

---

## Step 2 — Backend: Render

1. Push this repo to GitHub.
2. Go to <https://render.com> → **New → Blueprint** → connect the repo.
   Render reads `render.yaml` and creates the **eduboard-api** web service.
   *(Or: New → Web Service, set **Root Directory** = `server`, Build = `npm install`, Start = `npm start`.)*
3. In the service's **Environment**, set:
   | Key | Value |
   |-----|-------|
   | `MONGO_URI` | your Atlas connection string from Step 1 |
   | `CLIENT_URL` | your Vercel URL (fill in after Step 3), e.g. `https://eduboard.vercel.app` |
   | `JWT_SECRET` | auto-generated by the blueprint (or paste your own) |
   | `NODE_ENV` | `production` |
4. Deploy. When live you'll get a URL like `https://eduboard-api.onrender.com`.
   Test it: open `https://eduboard-api.onrender.com/api/health` → should return `{"status":"ok",...}`.

### Seed the production database (once)
From your local machine, point the seed at Atlas:
```bash
# server/.env  (temporarily)
MONGO_URI=<your atlas connection string>
```
```bash
npm run seed
```
This loads the demo courses, plans, and accounts into Atlas. Then revert your local `.env` back to localhost if you like.

> ⏳ Render's free tier sleeps after ~15 min idle, so the first request after a pause takes ~30s to wake. That's the cold start.

---

## Step 3 — Frontend: Vercel

1. Go to <https://vercel.com> → **Add New → Project** → import the repo.
2. Set **Root Directory** = `client`. Vercel auto-detects Vite (build `npm run build`, output `dist`).
3. Add an **Environment Variable**:
   | Key | Value |
   |-----|-------|
   | `VITE_API_URL` | your Render URL from Step 2, e.g. `https://eduboard-api.onrender.com` |

   ⚠️ No trailing slash, and **do not** include `/api` — the app appends it.
4. Deploy. You'll get a URL like `https://eduboard.vercel.app`.

---

## Step 4 — Connect the two (CORS)

Go back to **Render → eduboard-api → Environment** and set `CLIENT_URL` to your Vercel
URL (e.g. `https://eduboard.vercel.app`, no trailing slash). Save → Render redeploys.

This lets the API accept requests from your frontend. To allow multiple origins
(e.g. a custom domain too), comma-separate them: `https://eduboard.vercel.app,https://eduboard.com`.

> ℹ️ In addition to `CLIENT_URL`, the API (`server/src/app.js`) automatically allows this
> project's Vercel deployment URLs (hostnames ending in `-badivanas-projects.vercel.app`),
> so **preview deployments work without editing `CLIENT_URL`**. If you fork the project under
> a different Vercel account, update that suffix check in `app.js` to match your own.

## Troubleshooting

| Symptom | Fix |
|---------|-----|
| Login/courses fail, console shows **CORS error** | `CLIENT_URL` on Render must exactly match your Vercel origin (no trailing slash). |
| **Network error** / 404 on API calls | `VITE_API_URL` on Vercel is wrong — must be the Render base URL, no `/api`. Redeploy after changing env vars. |
| First request takes ~30s | Render free tier cold start — upgrade to a paid instance to keep it warm. |
| **MongoServerError: bad auth** | Wrong user/password in `MONGO_URI`, or special characters not URL-encoded. |
| Refreshing `/courses` 404s on Vercel | Ensured by `client/vercel.json` (SPA rewrite) — make sure root dir is `client`. |
| Page loads but no data | Seed the Atlas database (Step 2) and confirm `/api/health` works. |
